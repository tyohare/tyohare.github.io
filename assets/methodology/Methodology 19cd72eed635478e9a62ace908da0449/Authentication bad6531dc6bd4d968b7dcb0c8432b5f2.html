Authentication

https://owasp.org/Top10/A07_2021-Identification_and_Authentication_Failures/

Technology for authentication:

  • HTML Forms
    • Most common on websites. Sending credentials through request body.
  • Client SSL Certifications / smart cards
    • Asymmetric key crypto
    • Requires client to have valid SSL certificate
  • HTTP Basic/Digest
    • base 64 encoded
    • http://user:pass@www.example.com
    • Digest is when user credentials and URL are hashed together
  • NTLM Kerberos
    • Client and Server shared trusted DC

Authentication identifies users and confirms that they are who they say they are…

Insecure authentication implementation that causes issues.

Finding and exploiting authentication vulnerabilities…

  • Weak password requirements
    • Register weak password accounts
    • Look for password policy in documentation
  • Missing or insecure MFA
  • Improper restriction of authentication attempts
    • Look for lockout mechanisms on failed login attemps
    • See if lockout mechanism is susceptible to bypasses
  • Verbose error messages
    • Review changes in requests to identify correct or incorrect information
  • Vulnerable transmission of credentials
  • Insecure forgot password functionality
  • Cookies that aren’t tied to verification codes
  • Insecure storage of credentials

Prevention:

  • Wherever possible, utilized MFA
  • Change all default credentials
  • Always use an encrypted channel
  • Only POST requests should be used to to transmit credentials to the server
  • Stored credentials should be hashed and salted using secure cryptography
  • Implement NIST 800-63b effective password policy
  • Use identical or generic errors on failed login attempts
  • Use a simple password checker to motivate stronger passwords
  • Implement robust brute force protection
  • Audit authentication validation and verification logic for errors

Authentication Checks:

Username enumeration via different responses:

  • We can enumerate valid usernames in Burp suite by identifying responses with different lengths.
  • We can do the same with a valid username and passwords

2FA Bypass:

  • Assumes you have valid password
  • Test what happens if we drop the 2FA request in burp
    • Does it enforce the endpoint?
    • Does it let us in anyways?
  • Is the 24A code brute forcible?
  • Can you bypass 2FA spoofing request responses?

Password reset broken logic:

  • Look for temporary forgot password tokens
    • Can we change this to any value and change the user whose password is getting reset?
    • Look for this int he post requests.

Username enumeration via subtly different responses:

  • We can find valid usernames by checking for differences in responses
    • We can test by creating a valid account and testing how responses will be different that a non-existent user.
    • Response length may be a different length.
    • We can do a negative search in burp for invalid login responses as well.

Username enumeration via response timing:

  • Check changes in response milliseconds to determine valid or invalid passwords
  • Invalid usernames may lack increase in response timings but valid usernames with trying increasingly complex passwords may see a spike in response timing.

Broken brute-force protection, IP block:

  • With brute force lockout, we can cycle a known valid account to reset lockout counters.
    • For example, 2 Brute force guesses and 1 valid login. repeat.

Username enumeration via account lock

  • Use clusterbomb with list of users and passwords and try to lockout the accounts.
  • Valid usernames will trigger a lockout policy, invalid will not.
  • Check for differences in response for lockout text
  • When an account is locked out and you have a correct user/pass combo, may provide a different response from the others.

2FA broken logic:

  • If we can send 2FA requests for other users.
    • We can try to exploit this to brute force 2FA pins
