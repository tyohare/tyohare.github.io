Server-Side Template Injection (SSTI)

First discover SSTI:

  • We can see SSTI when our text is reflected back to us

$((<%[%'"}}%\
  • Confirming the template engine can be done in two ways
  1. Try to cause an error message.
  1. Attempt an engine-specific exploit.
  • Once we know the templating engine we will know the backend language…
  • Then we can use the language and knowledge of the templating engine to build a payload (Reference the docs!)
  • Sometimes it depends on what libraries are in use
  • Additionally you can try to utilize the global object class and build from there

Try the math equations to determine templating engine

Run SSTI related commands?

${{7*7}}  - Java, Expression Language
${7*7} - Java, Expression Language, Freemarker, Groovy, Mako, Thymeleaf 
{ {7*7} } - Jinja2, Pebble, Twig, Jinjava, HuBL, Nunjucks, Tornado 
{7*7} - Smarty
@(7*7) or @{7*7} - ASP.NET Razor
__${7*7}__::.x - Spring View Manipulation (Java)
__*{7*7}__::.x - Spring Framework (Java)
#{7*7} - Expression Language, Freemarker, Ruby, Jade, Pug
<%= 7*7 %> - Ruby
#set($x=7*7)${x} - Velocity

Mitigation:

  • Avoid passing unsanitized input to templates
  • Utilizing non-logic based templating engines
  • Deploy applications in sandboxed environment to limit SSTI impact

Empty expressions: ${} /£} / {£}} etc.

Divide by zero: ${1/0} / {1/0} / {{1/0}} etc.

Invalid variable names: ${wrong} /{wrong} /{{wrong}} etc.
