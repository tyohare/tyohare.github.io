Sessions

Send authenticated request to burp repeater

delete cookies one by one until app no longer returns the authenticated responses.

Set Cookie: header:

  • HttpOnly
    • prevents JS document.cookie web API from reading the cookie
    • Should be used on all cookies that contain sensitive information, especially session id cookies
    • Can be exploited by leveraging XSS to read cookie, then the attack sets the cookie in their browser…

  • Secure
    • Prevents the browser from sending a cookie in an unencrypted HTTP request
    • Can be exploited in MITM attacks to identify cookie

  • SameSite
    • Prevents browser from automatically sending a cookie to a site in requests that originate from another site
      • Strict - only permits same-site cookies be sent
      • Lax - same as strict, but permits cross-site cookies to be sent in GET requests when triggered by a top-level navigation action (Like clicking a link), not via POST or cross-origin frames
      • None - Cookies always sent cross-site

Alternatives to Sessions:

  • WWW-Authenticate response header (Basic, NTLM, Negotiate)
  • State mechanisms - ASP.NET ViewState
  • JWTs

Descriptive tokens (Could be changed…)

  • PHP - PHPSESSID
  • J2EE - JSESSIONID
  • ColdFusion - CFID/CFTOKEN
  • ASP.NET - SessionId

Session token disclosure vulnerabilities

  • Network disclosure due to lack of HTTPS
  • Log disclosure

Token to session mapping vulnerabilities

  • concurrent logins (multiple tokens)
  • Not checking the entire token is valid

Session termination

  • 15 min idle timeout recommended

Session Fixation

  • Is the same cookie reused on re-authentication?
  • We can just set the cookie in the request to gain access to the account
